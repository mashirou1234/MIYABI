# MIYABI Issue設計ガイド

最終更新: 2026-03-07

## 1. 目的

本書は、GitHub Issue を「完成へ近づくための実行可能な 1 タスク」に揃えるための正本とする。  
特に手動で整形した `codex:order` Issue と、Codex オートメーションで扱う `codex:queue` Issue は、本書の粒度・記載項目・受け入れ条件に従う。

目標:

- Issue を読めば、担当者や Codex が追加解釈なしで着手できる
- `PLAN.md` / 到達トラック / 設計書 / PR / 検証ログ が 1 本の導線で繋がる
- 「大きすぎる Issue」「曖昧な Issue」「完了条件が見えない Issue」を減らす

## 2. 正本の役割分担

Issue は単独で完結させず、必ず既存の正本に紐付ける。

| 知りたいこと | 正本 | Issue に必須で書くこと |
| --- | --- | --- |
| 何を先にやるべきか | `PLAN.md` | 対象フェーズ/タスク番号 |
| 今どのゲートを埋めるのか | `docs/CORE_DEVELOPMENT_TRACK.md` / `docs/GAME_DEVELOPMENT_TRACK.md` | 対象トラックとステージ |
| 何を正として実装するのか | 仕様書/設計書/運用書 | 参照ドキュメント |
| どこまでやれば閉じてよいか | Issue 本文 | 受け入れ基準 |
| 完了の証拠は何か | Issue コメント / PR 説明 / 実行ログ | 検証コマンドと期待結果 |

## 3. Issue 種別

### 3.1 Epic / 目標整理

- 用途:
  - 複数 Issue に分割すべき大きなテーマの整理
  - 直近 1 run で終わらないゴールの可視化
- 特徴:
  - 実装を直接回す単位ではない
  - `codex:queue` の対象にしない

### 3.2 Development Task / 実装タスク

- 用途:
  - 実装、ドキュメント更新、CI/テスト追加などを 1 テーマで完了させる
- 特徴:
  - Codex オートメーション対象の標準
  - `1 run 1 issue` を守る
  - 15〜45 分で閉じる粒度を目安にする

### 3.3 Decision / Investigation / 設計判断

- 用途:
  - 実装前に判断基準を固定する
  - 複数案の比較、境界条件、互換性判断、導入可否の整理
- 特徴:
  - 成果物は「設計判断」「評価表」「次 Task の切り出し」
  - 「調査して必要なら実装する」は禁止し、調査と実装を分離する

### 3.4 Bug

- 用途:
  - 不具合・クラッシュ・回帰の報告と修正
- 特徴:
  - `.github/ISSUE_TEMPLATE/bug_report.md` を使用する

## 4. `codex:queue` に入れてよい Task の条件

以下をすべて満たす Issue のみ、実装キューに入れる。

- 1 テーマである
- 対象トラック/ステージ/フェーズが 1 つに絞られている
- 作業範囲 In / Out が明記されている
- 受け入れ基準が観測可能な文で書かれている
- 最低 1 件の検証コマンドまたは確認シナリオがある
- 更新対象ドキュメントまたは記録先が明記されている
- ブロッカー条件または依存 Issue が分かる

逆に、次のような Issue は `codex:queue` に入れない。

- 例: 「C1 を達成する」「UI を改善する」「3D を進める」
- 例: 「調査して必要なら修正する」
- 例: 受け入れ基準が「よさそうならOK」になっている
- 例: コード、設計、配布、性能改善が 1 件に混在している

## 5. ラベル運用ルール

`codex:*` ラベルは「Issue の性質」と「現在位置」を分けて表現する。
特に `codex:order` と `codex:queue` は役割を混同しない。

### 5.1 `codex:order`

- 用途:
  - 手動レーンで、repo 文脈と skill を使って整形した開発 Issue
  - まだ自動レーンへ流す前段の Issue
- 特徴:
  - 実装方針、参照正本、受け入れ基準を人間が先に固定する
  - 大きめのテーマを分割する入口として使ってよい
  - 自動実行可能とは限らない

### 5.2 `codex:queue`

- 用途:
  - `scripts/orch.sh` の自動レーンに投入してよい Issue
- 特徴:
  - `1 run 1 issue` 前提
  - 本書「4. `codex:queue` に入れてよい Task の条件」を満たす必要がある
  - 15〜45 分で閉じる粒度を維持する

### 5.3 `codex:claimed`

- 用途:
  - 自動レーンが着手中であることを示す
- 特徴:
  - `codex:queue` から claim された Issue に付与する
  - 手動レーンでは原則使わない

### 5.4 `codex:blocked`

- 用途:
  - 自動レーンで完遂できず停止したことを示す
- 特徴:
  - Issue コメントに停止理由、暫定対応、次の 1 アクションを残す
  - `codex:claimed` は外し、必要なら `codex:order` に戻して手動再整理する

### 5.5 推奨ライフサイクル

1. 手動で Issue を起票・整形し、`codex:order` を付ける
2. 分割・具体化して自動レーン向きの粒度へ落とす
3. 自動投入可能になったら `codex:order` を外し、`codex:queue` を付ける
4. 自動レーンが開始したら `codex:claimed` を付ける
5. 完了時は `codex:claimed` を外して close する
6. 停止時は `codex:blocked` を付け、必要なら手動レーンで再整理する

補足:

- `codex:order` は「発注済み・整理済み」を表すラベルであり、優先順位そのものは表さない
- 実行順は Issue タイトル接頭辞（例: `[01]`）または Project の優先度項目で管理する
- `codex:order` と `codex:queue` は同時常用しない

## 6. 良い Task Issue の原則

### 6.1 粒度

- 1 Issue = 1 主成果物
- 1 Issue = 1 主な失敗理由
- 1 Issue = 1 PR で説明できる範囲
- 45 分を超える見込みなら先に分割する

### 6.2 曖昧語を避ける

避ける表現:

- 「整理する」
- 「改善する」
- 「対応する」
- 「必要なら」
- 「いい感じに」

推奨表現:

- 「`sample_game` 側へ `GameState` を移す」
- 「`cargo test --manifest-path logic/Cargo.toml` が通る」
- 「`docs/CORE_DEVELOPMENT_TRACK.md` から参照リンクを追加する」

### 6.3 受け入れ基準は観測可能に書く

悪い例:

- 描画が安定していること
- 設計が整理されていること

良い例:

- `docs/` 配下に境界ルール文書が存在する
- `sample_game` から `core/include/*` への直接参照が存在しない
- `ctest --test-dir build/core --output-on-failure` が成功する

### 6.4 検証は「誰でも同じ結果になる」形で書く

- コマンドを具体的に書く
- 手動確認なら操作手順と期待結果を書く
- Issue コメントまたは PR に残すログを先に決める

## 7. 推奨タイトル規約

厳密固定ではないが、以下の順を推奨する。

`[<Track>][<Gate or Phase>][Bench-80] <短い動詞句>`

例:

- `[Core][C1][Bench-80] sample_game 境界分離の次候補を docs に追記する`
- `[Game][G2][Bench-80] 30分連続プレイ検証手順を固定する`
- `[Core][C1][Bench-80] input mapping の公開API境界を定義する`

## 8. Development Task の必須項目

実装タスク Issue には、最低限以下を含める。

### 8.1 背景

- なぜ今これが必要か
- どの未達条件や不足に紐づくか

### 8.2 目的

- この Issue 単体で何を前進させるか
- 何を成果物として残すか

### 8.3 対象ゲート / 正本

- 対象トラック: `Core` または `Game`
- 対象ステージ/フェーズ: `C1`, `G2`, `Phase 10.1` など
- 参照ドキュメント: 1〜3 件に絞る

### 8.4 作業範囲 In / Out

- In:
  - 今回やること
- Out:
  - 今回やらないこと
  - 境界外のリファクタや別テーマ

### 8.5 受け入れ基準

- 3〜5 項目を目安にする
- 完了判定を読んだだけで close 可否が判断できる状態にする

### 8.6 検証

- コマンド、テスト、手動シナリオを明示する
- 期待結果を `PASS/FAIL` で判断できるようにする

### 8.7 更新対象

- `PLAN.md`
- `docs/CODEX_MIGRATION_STATUS.md`
- 到達判定に影響するなら `docs/CORE_DEVELOPMENT_TRACK.md` / `docs/GAME_DEVELOPMENT_TRACK.md`
- 仕様/設計変更があるなら該当設計書
- Issue コメントまたは PR へ残す実行結果

### 8.8 依存 / ブロッカー

- 先行 Issue が必要か
- 何が発生したら `codex:blocked` に切り替えるか

## 9. 分割ルール

以下のいずれかに当てはまる場合は分割する。

- 完了条件が 2 種類以上ある
- 失敗理由が 2 種類以上ある
- コア/ゲーム/SDK/CI など異なる責務が混在している
- 参照すべき正本が 4 件以上になる
- 手動検証だけで長時間を要する
- 実装前に設計判断が必要

分割例:

- 悪い例:
  - `C1 達成のために sample_game 分離と keybind と tilemap を進める`
- 良い例:
  - `sample_game へ GameState を移す`
  - `ButtonAction を sample_game 側ハンドラへ委譲する`
  - `入力マッピング設定の保存仕様を定義する`
  - `タイルマップ読込フォーマットを決める`

## 10. 現時点の優先 Issue 群

現在の到達状況を踏まえると、次の系統を優先して切り出すと完成に直結しやすい。

### 10.1 G2 到達向け

- 30分連続プレイ検証の手順固定
- 30分プレイ時のログ採取と結果記録
- 初回起動から終了までの導線確認
- 第三者による 1OS 配布再現確認

### 10.2 C1 到達向け

- `sample_game` / `logic` / `core` の責務再分離
- スプライトアニメーション最小導入
- タイルマップ/レベルデータ読込
- 入力マッピングと設定保存の接続
- 外部サンプル再利用 2 本の成立条件整理

### 10.3 運用品質向け

- Issue コメントに残す検証ログのフォーマット固定
- 破壊的変更時の告知テンプレート適用条件の明確化
- 設計判断 Issue と実装 Task Issue の分離運用

## 11. Development Task テンプレート

以下を標準テンプレートとして使う。

```md
## 背景
- <なぜ今必要か>

## 目的
- <このIssue単体で前進させること>

## 対象ゲート / 正本
- トラック: Core / Game
- ステージ or フェーズ: <C1 / G2 / Phase 10.1 など>
- 参照:
  - `PLAN.md`
  - `docs/...`

## 作業範囲 In
- <今回やること>

## 作業範囲 Out
- <今回やらないこと>

## 受け入れ基準
- [ ] <観測可能な完了条件 1>
- [ ] <観測可能な完了条件 2>
- [ ] <観測可能な完了条件 3>

## 検証
1. `<コマンド or 手順>`
2. `<コマンド or 手順>`

## 更新対象
- `PLAN.md`
- `docs/CODEX_MIGRATION_STATUS.md`
- `<必要なら該当設計書>`
- Issue コメントまたは PR に残す実行結果

## 依存 / ブロッカー
- 依存Issue: <なければ なし>
- blocked 条件: <該当時は codex:blocked へ切替>

## リスク
- <主なリスク>
```

## 12. Issue 作成前チェックリスト

- [ ] この Issue は 1 テーマに絞られている
- [ ] 対象トラック/ステージ/フェーズが 1 つに決まっている
- [ ] 参照ドキュメントが明記されている
- [ ] In / Out が書かれている
- [ ] 受け入れ基準が観測可能である
- [ ] 検証コマンドまたは手動手順がある
- [ ] 完了時に更新すべき資料が分かる
- [ ] ブロッカー条件が明記されている

このチェックを満たした Issue から `codex:queue` へ入れる。
