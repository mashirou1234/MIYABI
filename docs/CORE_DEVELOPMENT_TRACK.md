## 1. 目的

エンジンとしての基盤機能を段階的に完成させ、2D/3D開発を継続可能にする。

## 2. ステージ定義

### C0: Core Runtime（現状）

- FFI経由でゲーム更新/描画データ受け渡しが成立
- 2D描画/入力/音声/2D物理の最小統合が成立
- SDK v0.1 配布物が成立
- Core Save サブシステム定義が成立（未接続）

### C1: 2D Engine Baseline

- 2D制作向けの実運用機能を提供
- アセットパイプライン、テスト/CI、性能回帰監視を導入
- 2D開発が継続可能な開発基盤を成立

### C2: 3D Foundation

- 透視投影、3Dメッシュ、ライト、深度処理の最小導入
- 2D機能を破壊せず 3D基盤を共存させる

### C3: 3D Engine Baseline

- 3D制作向けの実運用機能を提供
- 表現拡張、性能最適化、3D向けツールを整備

### C4: Ecosystem Competitiveness（長期）

- 開発体験（デバッグ/拡張/ドキュメント）を整備
- マルチプラットフォーム運用と互換性方針を確立

## 5. ステージ別 完了定義（DoD）

各ステージは次の共通フォーマットで判定する。

- 完了定義:
  - 対象機能がステージ定義どおりに実装済み
  - 既存機能を壊さない最小回帰確認が完了
  - 関連ドキュメントが更新済み
- 判定エビデンス:
  - 実行ログ、テスト結果、またはスクリーンショット
  - 更新したドキュメント差分

## 6. 完了判定の定量基準

完了判定は「対象タスクのチェック完了率」「必須検証コマンド通過率」「DoDエビデンス記録数」で判定する。

- チェック完了率:
  - 対象ステージに紐づく `PLAN.md` のチェックボックスを 100% 完了（`[x]`）する
- 必須検証コマンド通過率:
  - 各ステージで定義した最小回帰コマンドを 100% 成功させる
- DoDエビデンス記録数:
  - 実行ログ 1 件以上
  - ドキュメント差分 1 件以上

### blocked 判定条件

次のいずれかを満たした時点で当該タスクは `blocked` と判定し、`docs/DEVELOPMENT_TRACK.md` の「依存解決ブロッカー記録欄」に同日中に1行追記する: (1) フェーズ開始条件や受け入れ基準を満たす前提情報が欠け、15分以内に補完できない、(2) 必須検証コマンドが環境・権限・外部依存で連続2回失敗し、ローカル修正で解消できない、(3) 上流仕様未確定またはレビュー待ちで45分以内に再開見込みが立たない。再開条件（誰が何を満たせば解除か）を同じ行の「暫定対応」に明記し、解消時に `resolved` へ更新する。
