# MIYABI 完成ロードマップ

最終更新: 2026-03-07

## 1. 目的

本書は、MIYABI を「いま動く試作」から「継続開発できる 2D/3D ゲームエンジン」へ進めるための実行順を固定する。
当面の完成定義は、`docs/GAME_DEVELOPMENT_TRACK.md` の `G2/G3` と `docs/CORE_DEVELOPMENT_TRACK.md` の `C1` を満たすこととする。

ただし、2D 完成は中間ゲートであり最終ゴールではない。
最終ゴールラインには `C2 -> C3 -> C4` と、`G4` に相当する 3D 基盤、3D 制作基盤、エコシステム競争力を含める。

## 2. 現在地

- コア現在地: `C0`
- ゲーム現在地: `G3`
- 直近ボトルネック:
  - `C1`: `sample_game` runtime boot path の最終反転と外部再利用証跡が未完
  - `C2/G4`: 3D 契約は固定済みだが、最初の実装 Task (`C2-01`, `C2-02`, `G4-01`) が未着手

## 3. 推進原則

- 完成直結の作業を優先し、文書微修正だけでキューを埋めない
- 手動で整形した開発 Issue は `codex:order` に置き、自動化可能になった粒度だけ `codex:queue` へ昇格する
- 1つのゲートで必要な証跡を先に決め、実装と同時にログを残す
- 大きいテーマは `15〜45分` で閉じる Task へ分割する

## 4. ロードマップ

### Wave 0: 開発促進レーンの是正

目的:
- 完成に効く Task が埋もれない運用へ切り替える

完了条件:
- `codex:order` / `codex:queue` の役割が固定されている
- open Issue を `完成直結` / `周辺整備` / `保留` に再分類できる
- G2/C1 の優先 Task が先頭から追える

### Wave 1: G2 到達

目的:
- 2D 縦切りを「完成」と呼べる検証状態にする

完了条件:
- 30分連続プレイで進行不能・クラッシュなし
- 初回起動から終了までの導線確認済み
- G2 証跡リンクを `docs/GAME_DEVELOPMENT_TRACK.md` に記録済み

### Wave 2: G3 到達

目的:
- 1OS 配布候補として第三者再現可能な状態にする

完了条件:
- 配布 ZIP の再現手順が第三者観点で確認済み
- 展開後起動と最小プレイ確認の証跡がある
- G3 証跡リンクを `docs/GAME_DEVELOPMENT_TRACK.md` に記録済み

### Wave 3: C1 到達

目的:
- 2D タイトルを継続開発できるエンジン基盤へ移行する

完了条件:
- `sample_game` がゲーム固有ロジックの受け皿として機能する
- `logic` が SDK コアへ寄り、ゲーム固有状態の残留が縮小している
- スプライトアニメーション、タイルマップ、入力マッピングの最小仕様が実装済み
- C1 証跡を `docs/CORE_DEVELOPMENT_TRACK.md` に記録できる

### Wave 4: 2D 再利用性の成立

目的:
- 単発サンプルではなく、エンジンとして再利用できる状態にする

完了条件:
- 外部サンプル 2 本以上で再利用できる
- リリース作業が再現可能
- API/ABI 更新ルールが継続運用できる

### Wave 5: C2 / 3D Foundation

目的:
- 既存 2D 基盤を壊さず、3D の最小レンダリング基盤を追加する

完了条件:
- 透視投影、3D メッシュ、ライト、深度処理が最小動作する
- 2D 回帰確認が通る

正本:
- `docs/CORE_3D_FOUNDATION_CONTRACT.md`

補足:
- 本 Wave 以降は「将来検討」ではなく、MIYABI の最終到達点に含める

### Wave 6: C3 / 3D Engine Baseline

目的:
- 3D タイトルを継続開発できる制作基盤へ進める

完了条件:
- 3D シーン制作の最小運用が成立する
- 性能計測と最適化の基準が定義される

正本:
- `docs/CORE_3D_PRODUCTION_BASELINE.md`

### Wave 7: C4 / Ecosystem Competitiveness

目的:
- 開発体験、拡張性、配布性の面で競争できる基盤を整える

完了条件:
- デバッグ、拡張、ドキュメント、互換性方針が継続運用可能
- 複数プラットフォーム運用の判断材料が揃う

正本:
- `docs/CORE_3D_PRODUCTION_BASELINE.md`

## 4.1 最終ゴールライン

本ロードマップで「完成」と呼ぶ状態は、短期と長期で次の 2 段階を持つ。

- 短期完成:
  - `G2/G3/C1` を満たし、2D ゲームを継続開発・配布できる
- 長期完成:
  - `C2/C3/C4` と `G4` に相当する 3D 基盤、3D 制作基盤、競争力強化まで到達する

したがって、3D は「後で考える項目」ではなく、2D 完成後に必ず接続する次のゴールラインとして扱う。

## 5. 直近 90 日の優先順

1. Wave 0:
   - open Issue の再分類方針を固定する
   - `codex:order` 起点の運用へ切り替える
2. Wave 1:
   - 30分プレイ検証手順を固定する
   - 30分プレイを実施し、致命的不具合の有無を記録する
   - 初回起動から終了までの導線を検証する
3. Wave 2:
   - 配布 ZIP の第三者再現チェックを行う
4. Wave 3:
   - `sample_game` へ `GameState` とプレイループを移す
   - `ButtonAction` と UI 遷移ハンドラを `sample_game` 側へ委譲する
   - 入力マッピング、アニメーション、タイルマップの最小仕様を順に埋める

## 6. `codex:order` へ置くべき Issue の条件

- 完成ゲート (`G2/G3/C1`) に直接寄与する
- まだ自動化に載せるには大きい、または判断が必要
- 受け入れ基準と検証方法は記述できる
- close 時に証跡 URL を残せる

## 7. このロードマップの使い方

1. 手動で起票する Issue は、まず本書のどの Wave を埋めるかを決める
2. Issue 本文に `対象Wave` と `対象ゲート` を書く
3. `codex:order` で運用し、十分に小さくなった Task だけ `codex:queue` へ送る
4. 各 Wave 完了時は、到達トラック正本と本書の両方に証跡を反映する
